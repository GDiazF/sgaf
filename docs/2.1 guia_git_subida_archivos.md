# Guía de Usuario: Subida de Cambios a GitHub

Este manual explica los pasos necesarios para enviar el código desde tu computadora local hacia el repositorio en GitHub.

## 1. El Ciclo de los 4 Pasos
Para que tus cambios lleven un control de versiones correcto, sigue este orden:

### Paso 1: Actualizar la versión (Opcional pero recomendado)
Este paso debe hacerse en la **Terminal de Windows** (PowerShell o Git Bash) dentro de tu PC local.

1. Abre la terminal en la carpeta raíz del proyecto.
2. Entra a la carpeta de frontend: `cd frontend`
3. Ejecuta el comando según el cambio realizado:
   - **Fix (Error menor)**: `npm run fix`
   - **Feat (Función nueva)**: `npm run feat`
   - **Release (Cambio mayor)**: `npm run release`

### Paso 2: Preparar los archivos
Este comando "marca" todos los archivos (incluyendo los de versión) para ser subidos.
```bash
git add .
```

### Paso 3: Comentar los cambios
Resume lo que hiciste en una frase.
```bash
git commit -m "v1.7.x: Descripción de lo que cambiaste"
```

### Paso 4: Enviar a GitHub (Push)
Sube físicamente los archivos a la nube.
```bash
git push origin HEAD
```

---

## 2. Comandos de Verificación (Útiles)

- **Ver qué archivos cambiaste**:
  `git status`
  *(Los archivos en rojo son cambios que aún no has agregado)*.

- **Ver el historial de lo que has subido**:
  `git log --oneline -n 5`

## 3. Resolución de Problemas Comunes

### Error: "The upstream branch of your current branch does not match"
Esto ocurre cuando tu rama local tiene un nombre distinto a la del servidor o no están vinculadas.
**Solución rápida**: 
```bash
git push origin HEAD
```
**Solución definitiva** (para que no vuelva a preguntar):
```bash
git push -u origin [nombre_de_tu_rama]
```

### Error: "Updates were rejected / non-fast-forward"
Esto sucede si alguien más subió cambios y tú no los tienes en tu PC. 
**Solución**: Baja los cambios primero:
```bash
git pull origin [nombre_de_tu_rama]
```

---
> [!TIP]
> Recuerda que primero debes actualizar la versión con `npm run fix` (o el que corresponda) antes de hacer el `git add .` si quieres que el cambio de versión también se suba.

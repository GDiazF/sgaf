# Ciclo de Vida y Gestión de Versiones - SGAF

Este documento define cómo actualizar el sistema y cómo manejar la numeración de versiones para mantener un historial profesional.

## 1. El Flujo de Trabajo (Git Flow)

NUNCA modifiques código directamente en el servidor de producción. Sigue estos pasos:

1. **Local**: Desarrollar el cambio en la PC de Windows.
2. **Prueba**: Verificar que funcione correctamente en modo desarrollo.
3. **Versionar**: Incrementar el número de versión (ver sección 2).
4. **Push**: Subir los cambios a GitHub (`git push`).
5. **Pull**: En el servidor Ubuntu, descargar los cambios (`git pull`).
6. **Despliegue**: Reconstruir los contenedores:
   ```bash
   docker compose up -d --build
   ```

## 2. Definición de Versiones (SemVer)
Seguimos el estándar **X.Y.Z**:

- **X (Mayor)**: Cambios que rompen compatibilidad o rediseños totales.
- **Y (Menor)**: Nuevas funciones o módulos completos (ej. Monitor de Red).
- **Z (Parche)**: Corrección de errores (fixes), parches de seguridad o cambios estéticos menores.

### Ubicación de la versión en el código:
Para subir de versión, debes actualizar estos archivos:
1. `frontend/src/version.js`: (Es lo que ve el usuario en la pantalla).
2. `frontend/package.json`: (Es el estándar técnico de Node.js).

## 3. Ejemplo de Actualización
Si hoy arreglamos un botón que no funcionaba (Fix):
- La versión pasa de `1.7.1` a `1.7.2`.

Si mañana agregamos el módulo de PING de routers (Feature):
- La versión pasa de `1.7.2` a **`1.8.0`**.
